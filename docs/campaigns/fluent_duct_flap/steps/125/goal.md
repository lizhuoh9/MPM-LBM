commit with the current code commit:

- `docs/current/ACTIVE_CAMPAIGN.json` records only
  `base_commit = 516b1aaa4c71d5468ce5ea444a21ffa07741c8bc`.
- The committed Step121 campaign manifest still has
  `git_commit = 516b1aaa4c71d5468ce5ea444a21ffa07741c8bc`, inherited from the
  earlier smoke state.

This is acceptable for the old smoke artifact but not acceptable before real
48^3 rows are produced. Future references, candidates, selected 96^3 duct, and

The patch must:

1. Split campaign identity from code identity.
   - Preserve the original campaign base commit as
     `campaign_base_commit`.
   - Record the code commit used for the current committed campaign metadata as
     `current_code_commit`.
   - Keep `base_commit` only as a backward-compatible alias for the campaign
     base while existing readers still use it.

2. Upgrade campaign manifest provenance.
   - Add `campaign_base_commit`.
   - Add `current_code_commit`.
   - Keep `git_commit` as a compatibility alias for `current_code_commit` in
     newly written manifests.
   - Add `phase_commit_history`, recording at least the phase name and code
     commit for every manifest-writing phase.
   - Preserve legacy `phase_history` for compatibility.
   - Preserve existing expected rows when a new phase appends to an existing
     manifest.

3. Record row-level code identity for future real rows.
   - Add `code_commit_at_run` to Step120 summary rows.
   - Add `code_commit_at_run` to Step120 row metadata.
   - This field must be generated by the runner at row creation time; it must
     not be hardcoded to a stale campaign commit.

4. Update current committed artifacts and docs.
   - Update `docs/current/ACTIVE_CAMPAIGN.json` with explicit
     `campaign_base_commit` and `current_code_commit`.
   - Update the committed Step121 campaign manifest under
     `outputs/step121_lbm_boundary_real_campaign_and_gate_correction/` to the
     new manifest schema, without claiming new real 48^3 or 96^3 execution.
   - Add a Step125 report documenting exactly what changed, what tests were
     run, and what remains blocked until real 48^3 execution.
   - Update current reading docs only if needed to point to Step125 as the
     latest goal/report.

## Important Commit-Identity Nuance

A JSON file committed into Git cannot stably contain the hash of the commit
that contains that same JSON file; amending the JSON changes the commit hash.

Therefore Step125 must use two different checks:

- Committed campaign docs/artifacts must record an auditable code commit that
  is an ancestor of the current repository `HEAD` and is newer than or equal to
  the campaign base. For this Step125 patch, that reviewed code commit is
  `26d4d0a9597a440272e2d479decbf1fbf95cb2ff`.
- Runtime-generated manifests must set `current_code_commit` and compatibility
  `git_commit` to the live `git rev-parse HEAD` at generation time.

Tests must reflect this distinction so the repository does not require an
impossible self-referential commit hash.
