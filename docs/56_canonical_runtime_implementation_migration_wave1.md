# Step 56 Canonical Runtime Implementation Migration Wave 1

Step 56 migrates the first leaf batch of runtime implementations into the canonical package tree created by Step 55.

## Scope

The migrated implementation owners are:

- `src/mpm_lbm/sim/lbm/config.py`
- `src/mpm_lbm/sim/lbm/relaxation_semantics.py`
- `src/mpm_lbm/sim/lbm/fluid.py`
- `src/mpm_lbm/sim/mpm/config.py`
- `src/mpm_lbm/sim/mpm/solid.py`
- `src/mpm_lbm/sim/units/mapper.py`
- `src/mpm_lbm/sim/coupling/projection.py`
- `src/mpm_lbm/sim/coupling/penalty.py`
- `src/mpm_lbm/sim/coupling/moving_boundary.py`

The corresponding legacy root files are now compatibility shims that import from those canonical modules.

## Dependency Direction

The accepted direction after this step is:

```text
legacy root import -> compatibility shim -> canonical implementation
```

The rejected direction is:

```text
canonical implementation -> lazy wrapper -> legacy root implementation
```

## Non-Goals

Step 56 does not migrate `fsi_config`, `fsi_driver`, `sim_config`, link-area modules, geometry modules, squid modules, wall-velocity modules, runtime geometry modules, or diagnostic geometry modules.

Step 56 also does not add any `48^3` link-area run, longer cycle, `64^3` run, LBM tau migration, changed historical physics formula, default solver behavior change, `external/taichi_LBM3D` edit, or `data/real_geometry_candidates` edit.

## Interpretation

Passing Step 56 means the first runtime ownership wave is now canonical-package owned, legacy imports remain compatible, lightweight behavior-preservation checks remain green, and Step 55 layout/import boundaries still hold.

Passing Step 56 does not mean real squid validation, jet validation, grid convergence, production readiness, or new FSI physics has been established.
